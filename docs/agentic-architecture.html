1. Agent là gì? So sánh với LLM thường

LLM thường

User: "Tìm sách Python"

LLM: "Tôi không biết DB của bạn
      có sách gì, bạn cho tôi
      biết đi."

❌ Không tự tìm được
❌ Chỉ dùng kiến thức training

Agent (tool-use)

User: "Tìm sách Python"

LLM: [gọi search_books("Python")]
Tool: [{"id": 2, "title": "Learning Python"}]
LLM: "Tôi tìm thấy cuốn
      'Learning Python' (id=2)"

✓ Tự động tìm trong DB thật
✓ Kết quả chính xác, up-to-date

Định nghĩa đơn giản

Agent = LLM + khả năng gọi functions (tools) + vòng lặp tự chủ. Thay vì trả lời ngay từ training data, agent thu thập thông tin từ tools rồi mới tổng hợp trả lời.

3. Tool-Use Loop — Trái tim của Agent

Ví dụ cụ thể: "Tìm sách Python và tóm tắt"

── Iteration 1 ───────────────────────────────────────────────── messages = [system, user("Tìm sách Python và tóm tắt")] → LLM: finish_reason = "tool_calls" tool_calls = [search_books(query="Python")] → execute_tools_parallel: search_books("Python") → '{"found":1,"books":[{"id":2,"title":"Learning Python",...}]}' messages.append(assistant) # assistant message với tool_calls messages.append(tool_result) # {"role":"tool","content":"..."} ── Iteration 2 ───────────────────────────────────────────────── messages = [..., assistant_1, tool_result_1] → LLM: finish_reason = "tool_calls" tool_calls = [summarize_book(book_id=2)] → execute_tools_parallel: summarize_book(2) → '{"summary":"Sách dạy Python từ cơ bản..."}' messages.append(assistant_2) messages.append(tool_result_2) ── Iteration 3 ───────────────────────────────────────────────── messages = [..., assistant_1, tool_1, assistant_2, tool_2] → LLM: finish_reason = "stop" content = "Tôi tìm thấy 'Learning Python'. Nội dung chính: ..." → RETURN answer # vòng lặp kết thúc

Code: _tool_use_loop() trong base_agent.py

while iteration < self.max_iterations:
    iteration += 1

    # 1. Gọi LLM với messages + tool schemas
    response = await self._call_llm(messages, tool_schemas)
    choice = response["choices"][0]
    finish_reason = choice["finish_reason"]

    # 2. LLM trả lời xong → return
    if finish_reason == "stop":
        return choice["message"]["content"]

    # 3. LLM muốn gọi tool → thực thi rồi loop
    if finish_reason == "tool_calls":
        tool_calls = choice["message"]["tool_calls"]
        messages.append(choice["message"])      # giữ assistant message
        results = await execute_tools_parallel(tool_calls, self.tools)
        messages.extend(results)                  # thêm kết quả tools
        continue                              # LLM đọc lại, quyết định tiếp

⚠️ Tại sao phải giữ assistant message? OpenAI yêu cầu: message có tool_calls phải có tương ứng tool result với cùng tool_call_id. Không có → API lỗi 400.

4. Tool Registry — Open/Closed Principle

Không có registry

# Phải sửa agent mỗi khi thêm tool
if tool_name == "search_books":
    result = search_books(...)
elif tool_name == "get_book":
    result = get_book(...)
elif tool_name == "NEW_TOOL":  # ← sửa đây
    ...

Có registry

# Thêm tool → không cần sửa agent
@register_tool(name="new_tool", ...)
async def new_tool(...) -> str:
    ...

# agent.py không thay đổi
result = TOOL_REGISTRY[tool_name]["fn"](**args)

TOOL_REGISTRY structure

TOOL_REGISTRY = {
    "search_books": {
        "fn":     <async function search_books>,
        "schema": {
            "type": "function",
            "function": {
                "name":        "search_books",
                "description": "Tìm kiếm sách theo từ khóa...",
                "parameters":  { /* JSON Schema */ }
            }
        }
    },
    ...
}

💡 LLM nhận schema (biết tool tên gì, args gì). Agent nhận fn để thực thi khi LLM yêu cầu.

5. Parallel Tool Execution

Sequential (chậm)

result_1 = await search_books(...)  # 200ms
result_2 = await get_book(...)       # 50ms
# Total: 250ms

Parallel với asyncio.gather (nhanh)

results = await asyncio.gather(
    _run_one(tc_1),   # search_books
    _run_one(tc_2),   # get_book
)
# Total: max(200ms, 50ms) = 200ms

LLM có thể quyết định gọi nhiều tools cùng lúc trong một round. asyncio.gather chạy tất cả song song, chờ hết rồi trả về.

6. Template Method Pattern — BaseAgent vs BookAgent

# base_agent.py — định nghĩa "khung thuật toán"
class BaseAgent(ABC):
    @abstractmethod
    def system_prompt(self) -> str: ...    # subclass override

    @abstractmethod
    def tools(self) -> list[str]: ...        # subclass override

    async def chat(self, user_message: str):
        messages = [system_prompt, user_message]
        return await self._tool_use_loop(messages, ...)  # cố định

# book_agent.py — chỉ cần điền vào chỗ trống
class BookAgent(BaseAgent):
    system_prompt = "Bạn là thư viện viên AI..."
    tools = ["search_books", "get_book", "summarize_book"]
    # KHÔNG cần viết lại loop — kế thừa từ BaseAgent

💡 Thêm GeraiAgent mới: chỉ cần tạo class với system_prompt và tools khác. Phần loop không thay đổi.

7. So sánh với vuonglearning (production system)

Feature book-management (học) vuonglearning (production)
Base class BaseAgent — abstract + chat() BaseAgent — abstract + execute() → streaming bytes
Response JSON text (sync wait for full answer) SSE stream (token-by-token, real-time)
Tool registry Manual @register_tool + __init__.py imports discover_tools() auto-imports from src/tools/
Parallel execution asyncio.gather asyncio.gather + dedup check
Tool definitions 3 tools (search, get, summarize) 20+ tools (web search, image gen, memory, calculator...)
Tracing Structured logging (logger.info) Langfuse spans + metrics (Prometheus)
Max iterations 8 (hard-coded) Configurable per agent, typically 5
LangGraph? Không — custom loop Không — custom loop for SSE control

8. Tại sao KHÔNG dùng LangGraph?

LangGraph phù hợp khi

  • Workflow phức tạp với nhiều nhánh conditional
  • Nhiều agents tương tác nhau (multi-agent)
  • Cần visualize graph
  • Không cần full control output stream

Custom loop phù hợp khi

  • Cần control từng byte của output (SSE streaming)
  • Tool-use đơn giản, không cần graph phức tạp
  • Muốn minimize dependencies
  • Cần tích hợp chặt với infra hiện có (Redis, metrics)
